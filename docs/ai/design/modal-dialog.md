# Modal Dialog Design

현재 화면을 잠시 막고 확인 또는 짧은 보조 입력을 처리하는 `<dialog>` 기준이다. 목록을 계속 조작해야 하는 상세는 `side-drawer.md`를 사용한다.

## 선택 기준

| 상황 | UI |
|---|---|
| 저장·취소·삭제·중지 실행 전 확인 | 확인 모달 |
| 작은 범위 보조 입력 | 등록/수정 모달 |
| 상세에서 정정·재검수 입력으로 전환 | 작업 전환 모달 |
| 행을 바꿔 가며 상세 조회 | 오른쪽 드로어 |
| 실행 후 짧은 결과 | 토스트 |

긴 업무 전체를 모달에 넣지 않는다.

## 공통 구조

```text
dialog.part-modal.[type]
- form 또는 div.part-modal-card
  - part-modal-header
    - h2
    - 짧은 설명
    - modal-close-button
  - modal-body
  - field-message 선택
  - part-modal-actions
```

- 본문만 내부 스크롤하고 페이지 높이를 늘리지 않는다.
- 저장 중 주요 행동과 닫기·취소를 비활성화한다.
- 실패는 모달 안에 짧게 표시한다.
- 공통 class 이름을 바꾸면 HTML·CSS·문서를 한 번에 변경한다.

## 확인 모달

- 사용자가 마지막으로 확인할 값 3~5개만 둔다.
- 수량이 중요한 업무는 총수량을 표시한다.
- 위험 행동은 `삭제`, `전표 취소`, `사용 중지`처럼 구체적으로 쓴다.
- 닫기·배경 클릭·취소는 API를 호출하지 않는다.
- API는 최종 확인 버튼에서만 호출한다.

## 등록/수정 모달

- 현재 화면 상태를 유지해야 하는 작은 입력에 사용한다.
- 필수 입력을 최소화하고 필드가 많으면 별도 페이지로 분리한다.
- 성공 후 목록·선택·요약을 실제 API 응답으로 갱신한다.
- API 없이 화면에만 임시 데이터를 추가하지 않는다.

## 작업 전환 모달

- 처음에는 읽기 전용 상세를 보여준다.
- 후속 행동을 선택하면 같은 모달의 입력 폼으로 전환한다.
- 취소하면 상세 상태로 돌아간다.
- 저장 성공 후 목록과 최신 상세를 다시 조회한다.

## 반응형

- 확인 모달은 약 520px, 일반 입력은 680px 이하를 우선한다.
- 높이는 viewport 여백 안에 두고 본문만 스크롤한다.
- 모바일은 좌우 12~16px 여백을 유지하며 필드와 버튼을 1열 또는 2열로 바꾼다.

## 접근성

- 제목 id와 `aria-labelledby`를 연결한다.
- 닫기 버튼에 `aria-label="닫기"`를 둔다.
- 상태 메시지는 필요하면 `role="status"`, `aria-live="polite"`를 사용한다.
- 저장 중 Escape·배경 클릭으로 중복 상태를 만들지 않는다.

## 금지

- 업무마다 다른 모달 뼈대
- 긴 도움말과 중첩 카드
- 단순 필터 변경 확인
- 확인 전에 위험 API 호출
- 반복 상세 조회에 모달 사용
