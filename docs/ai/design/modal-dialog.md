# Modal Dialog Design

모달은 현재 화면의 맥락을 유지한 채 확인, 보조 입력, 짧은 작업 전환을 처리하는 UI다.

PCS의 모달은 업무마다 내용은 달라도 공통 뼈대는 같아야 한다. 새 업무 모달을 만들 때는 공통 구조를 유지하고, 필요한 경우 유형별 본문 양식과 보조 클래스로만 확장한다.

## 공통 원칙

- HTML은 기본적으로 `<dialog>`를 사용한다.
- 공통 구조는 `part-modal`, `part-modal-card`, `part-modal-header`, 본문 영역, 액션 영역을 따른다.
- 업무별 클래스는 폭, 내부 스크롤, 본문 간격처럼 레이아웃 보정을 위해서만 추가한다.
- 모달은 버튼 클릭처럼 사용자가 명시적으로 요청한 일시 작업에 사용한다.
- 목록 행을 빠르게 바꿔 가며 확인하는 상세 조회는 모달이 아니라 오른쪽 상세 슬라이드 패널을 우선 사용한다.
- 목록을 보면서 항상 열어두는 입력은 `form-panel.md`를 우선 사용한다.
- 긴 업무 전체를 모달 안에 넣지 않는다.
- 실행 전 확인은 모달, 실행 후 짧은 결과 알림은 토스트를 우선 사용한다.

## 공통 구조

기본 뼈대:

```text
dialog.part-modal.[type-modal]
- .part-modal-card
  - .part-modal-header
    - h2
    - 짧은 보조 문장 또는 요약 값
    - button.modal-close-button
  - .modal-body 또는 업무별 본문 컨테이너
  - .part-modal-actions 또는 업무별 액션 컨테이너
```

현재 공통 클래스명은 `part-modal`을 사용한다. 이름은 부품 빠른 등록에서 시작했지만 입고 확인, 검수 작업 전환 등 공통 모달 구조에도 사용한다. 추후 `modal-card` 같은 이름으로 바꾸려면 HTML, CSS, MD를 한 번에 일괄 변경한다.

공통 기준:

- 제목은 모달에서 사용자가 할 일을 직접적으로 말한다.
- 보조 문장은 한 문장 또는 짧은 요약 값으로 제한한다.
- 닫기 버튼은 우상단에 둔다.
- 본문은 모달 내부에서 스크롤되게 하고, 페이지 전체 높이를 늘리지 않는다.
- 저장 중에는 주요 액션과 닫기/취소 버튼을 비활성화한다.
- 저장 실패 시 모달 안에 짧은 오류 메시지를 표시한다.

## 모달 유형

### 확인 모달

저장, 취소, 삭제, 중지처럼 실행 전 확인이 필요한 작업에 사용한다.

예시:

- 입고 전표 저장 확인
- 입고 전표 취소 확인
- 검수 결과 저장 확인
- 템플릿, 항목, 선택지 사용 중지 확인
- 삭제 확인

구조:

```text
dialog.part-modal.confirm-modal
- form.part-modal-card
  - .part-modal-header
  - .confirm-summary
  - .modal-message 또는 .field-message
  - .part-modal-actions
```

본문 기준:

- 사용자가 마지막으로 확인해야 하는 핵심 값만 둔다.
- 요약 항목은 3~5개 이하로 유지한다.
- 입고/출고처럼 수량이 중요한 작업은 총 수량을 반드시 표시한다.
- 위험 작업은 버튼 문구를 `삭제`, `취소 처리`, `사용 중지`처럼 구체적으로 쓴다.
- 위험 작업의 주요 액션은 가능하면 `btn-danger` 같은 destructive 스타일을 사용한다.
- 닫기, 배경 클릭, 하단 취소는 API를 호출하지 않는다.

입고 전표 취소 예:

```text
전표를 취소할까요?
취소 후 원본 전표와 재고 이력은 보존되고, 입고 재고만 되돌립니다.

전표번호
거래처
부품 수량

[취소] [전표 취소]
```

### 조회 상세와 슬라이드 구분

조회 상세는 사용 방식에 따라 모달과 슬라이드 패널을 구분한다.

오른쪽 상세 슬라이드 패널을 우선 사용하는 경우:

- 전표 통합 조회처럼 전표 행을 계속 바꿔 보며 상세를 확인하는 화면
- 검수 이력처럼 관리번호 행을 계속 바꿔 보며 이력을 확인하는 화면
- 상세를 열어 둔 채 목록에서 다른 행을 바로 클릭해야 하는 화면

모달을 사용할 수 있는 경우:

- 전표 취소 전 상세 요약처럼 확인 작업에 종속된 짧은 조회
- 검수 관리 화면 안에서 정정/재검수 입력으로 바로 이어지는 작업 전환
- 현재 화면의 목록 조작을 잠시 막아도 되는 일시 작업

예시:

- 전표 취소 전 상세 요약
- 상태 변경 이력 상세

구조:

```text
dialog.part-modal.[domain-detail-modal]
- .part-modal-card
  - .part-modal-header
  - .modal-detail-body 또는 업무별 상세 컨테이너
    - 대표 식별값 카드
    - 주요 메타 정보
    - 상세 항목 목록
  - .part-modal-actions 또는 업무별 후속 작업 버튼
```

모달 조회 본문 기준:

- 첫 화면은 읽기 전용이어야 한다.
- 대표 식별값은 첫 번째 카드나 상단 블록에 크게 둔다.
- 메타 정보는 2열 안팎으로 짧게 배치한다.
- 반복 항목은 카드 또는 리스트로 나누되 과한 중첩 카드는 피한다.
- 후속 작업이 있으면 본문 하단에 버튼을 둔다.
- 확인 작업에 종속된 짧은 상세 모달은 내부 스크롤을 사용한다.

검수 관리 내 이력 작업 전환 예:

```text
검수 이력 상세
3개 항목

관리번호
정정 / B / 통과

입고 전표
품목
검수일
처리자
판매상태
메모
이력 관계

항목별 결과

[정정 등록]
[재검수 등록]
```

상세 슬라이드 패널 기준:

- `<dialog>`를 사용하지 않는다.
- `aria-modal="true"`를 사용하지 않는다.
- 배경 오버레이를 깔지 않는다.
- 패널이 열린 상태에서 다른 행을 클릭하면 상세 내용만 교체한다.
- 행과 패널 내부를 제외한 영역을 클릭하면 패널을 닫는다.
- 닫기 버튼과 `Escape` 키 닫기를 제공한다.

### 등록/수정 모달

현재 화면을 유지하면서 작은 범위의 값을 입력하거나 수정할 때 사용한다.

예시:

- 빠른 기준 데이터 등록
- 품목 상세입력
- 반복 하위 항목 입력
- 작은 범위의 정보 수정

구조:

```text
dialog.part-modal.[edit-modal]
- form.part-modal-card
  - .part-modal-header
  - .part-modal-fields
  - .field-message
  - .part-modal-actions
```

본문 기준:

- 필수 입력을 최소화한다.
- 입력 필드가 많으면 접힘 영역 또는 별도 화면으로 분리한다.
- 저장 후 현재 화면의 목록, 선택값, 요약을 즉시 갱신한다.
- 등록 API가 아직 없으면 화면에만 임시 데이터를 추가하지 않는다.
- 전체 등록 페이지 수준의 긴 업무는 모달이 아니라 본문 폼이나 별도 페이지로 둔다.

### 작업 전환 모달

조회 상세에서 후속 작업 입력으로 전환되는 모달이다. 첫 화면은 조회 상세이고, 버튼을 누르면 같은 모달 안에서 보조 입력 폼을 표시한다.

예시:

- 검수 이력 상세에서 정정 등록
- 검수 이력 상세에서 재검수 등록

구조:

```text
dialog.part-modal.[domain-detail-modal]
- .part-modal-card
  - .part-modal-header
  - .modal-detail-body
    - 읽기 전용 상세
    - 후속 작업 버튼
    - 작업 입력 폼(hidden 상태로 시작)
```

전환 기준:

- 처음 열렸을 때는 상세 조회 상태로 시작한다.
- 후속 작업 버튼을 누르면 작업 선택 버튼은 숨기고 입력 폼을 보여준다.
- 입력 폼에는 기준 이력 ID나 기준 정보를 짧게 표시한다.
- 취소하면 입력 폼을 닫고 상세 조회 상태로 돌아간다.
- 저장 성공 후 목록을 갱신하고 최신 상세를 다시 조회한다.

## 업무별 적용

| 업무 | 모달 유형 | 기준 |
|---|---|---|
| 입고 전표 저장 | 확인 모달 | 저장 전 요약 확인 |
| 입고 전표 취소 | 확인 모달 | 위험 작업, 취소 API는 확인 버튼에서만 호출 |
| 입고/출고 전표 상세 | 상세 슬라이드 | 목록 클릭을 막지 않고 상세 확인 |
| 검수 결과 저장 | 확인 모달 | 관리번호와 결과 요약 확인 |
| 검수 관리 내 이력 상세 | 작업 전환 모달 | 읽기 전용 상세 + 정정/재검수 전환 |
| 검수 이력 화면 관리번호 상세 | 상세 슬라이드 | 행을 바꿔 가며 이력 확인 |
| 검수 정정/재검수 | 작업 전환 모달 | 상세에서 후속 입력 폼으로 전환 |
| 검수 템플릿 선택지 비활성화 | 확인 모달 | 사용 중지 전 확인 |
| 품목 상세입력 | 등록/수정 모달 | 현재 품목 폼 상태 유지 |
| 빠른 기준 데이터 등록 | 등록/수정 모달 | 작은 범위의 보조 입력 |

## 반응형 기준

데스크톱:

- 화면 중앙에 둔다.
- 확인 모달은 520px 전후로 둔다.
- 일반 등록/수정 모달은 680px 전후를 넘지 않는다.
- 확인 작업에 종속된 짧은 상세 모달은 600~680px 범위를 우선 사용한다.
- 모달 높이는 `calc(100vh - 여백)` 안에 두고 본문만 스크롤한다.

모바일:

- 폭은 `calc(100vw - 24px)` 또는 `calc(100vw - 32px)`로 둔다.
- 본문 입력과 메타 정보는 1열로 전환한다.
- 하단 버튼은 세로 1열 또는 동일 폭 2열로 배치한다.
- 텍스트가 버튼 안에서 줄바꿈되어도 높이가 안정적으로 유지되게 한다.

## 접근성 기준

- `<dialog>`에 `aria-labelledby`를 지정한다.
- 제목 `h2`의 id와 `aria-labelledby`를 연결한다.
- 닫기 버튼에는 `aria-label="닫기"`를 둔다.
- 상태 메시지는 필요하면 `role="status"`와 `aria-live="polite"`를 사용한다.
- `Escape`와 배경 클릭으로 닫을 수 있는지는 작업 위험도에 따라 결정한다.
- 저장 중에는 배경 클릭 또는 닫기 버튼으로 중복 제출 상태를 만들지 않는다.

## 금지

- 업무별로 완전히 다른 모달 뼈대를 만들지 않는다.
- 카드 안에 또 모달처럼 보이는 카드를 중첩하지 않는다.
- 모달 안에 긴 도움말을 넣지 않는다.
- 단순 필터 변경마다 확인 모달을 띄우지 않는다.
- 저장 확인 없이 API를 먼저 호출하지 않는다.
- API 없이 화면에만 임시 등록 데이터를 추가하지 않는다.
- 목록 행 상세를 반복 확인해야 하는 화면에 모달을 사용하지 않는다.
