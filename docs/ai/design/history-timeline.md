# History Timeline Design

관리번호의 입고·검수·상태 변경·출고 과정을 추적하는 이력 표현 기준이다. API와 저장 의미는 해당 feature 문서가 원본이다.

## 원칙

- 관리번호 기준으로 추적할 수 있어야 한다.
- 누가, 언제, 무엇을, 왜 변경했는지 보여준다.
- 취소·정정·재검수도 별도 이력으로 남긴다.
- 저장된 사실과 화면의 추론을 구분한다.

## 구조

```text
history filter
history summary
timeline 또는 history table
```

여러 대상 검색은 `data-table.md`, 관리번호 반복 상세는 `side-drawer.md`를 함께 사용한다.

## Filter

- 관리번호, 기간, 이력 종류, 처리자, 분류, 상태처럼 실제 조회 가능한 조건만 둔다.
- 추상적인 질문형 필터명을 사용하지 않는다.
- 누적 이력은 기본 기간 조건과 서버 페이징을 제공한다.

## Timeline item

```text
이력 종류
관리번호·품목
before -> after
처리자·처리 시각
사유·메모
연결 전표 또는 검수 ID
```

- 상태 변화는 before → after로 표시한다.
- 중요한 상태는 공통 badge를 사용한다.
- 긴 메모는 접거나 상세로 보낸다.
- 다음 행동이 명확하면 해당 단계 안에 한 번만 표시한다.

## 전체 목록과 개별 timeline

- 전체 목록은 여러 관리번호의 이력을 검색한다.
- 개별 timeline은 입고부터 현재까지 시간순으로 표시한다.
- 현재 상태 요약은 개별 timeline 상단에 둔다.
- 현재 상태 설명 문장은 저장된 이력으로 확인 가능한 사실만 사용한다.

## 반응형

- 모바일은 1열 카드다.
- before/after는 줄바꿈되어도 의미가 유지되어야 한다.
- 관리번호는 카드 상단에서 쉽게 찾을 수 있어야 한다.
- 필터는 1열로 전환한다.
