# History Timeline Design

이력 관리와 개별 부품 타임라인 화면 기준이다.

## 적용 대상

- `/w/{companyCode}/history`
- 개별 부품 전체 타임라인
- 입출고 이력
- 검수 이력
- 상태 변경 이력
- 부품 상세 안의 이력 탭

## 적용 제외

- 일반 목록 테이블
- 대시보드 요약 통계
- 단순 등록/수정 패널

## 목적

PCS에서 이력 관리는 핵심 기능이다.

이력 화면은 단순히 로그를 나열하는 화면이 아니라, 현재 상태가 어떤 과정을 거쳐 만들어졌는지 추적하게 해야 한다.

## 기본 원칙

- 관리번호 기준으로 추적할 수 있어야 한다.
- 입고, 검수, 상태 변경, 출고가 하나의 흐름으로 이어져야 한다.
- 누가, 언제, 어떤 이유로 변경했는지 보여야 한다.
- 삭제된 것처럼 보이는 이력은 없어야 한다.
- 취소/정정/재검수도 별도 이력으로 남긴다.

## 기본 구조

```text
workspace-layout
- workspace-main
  - workspace-header
  - history-filter
  - history-summary
  - timeline 또는 history-table
```

## 이력 필터

필터는 사용자가 추적하려는 질문을 빠르게 만들 수 있어야 한다.

필터 예:

- 관리번호
- 기간
- 이력 종류
- 처리자
- 품목 분류
- 상태

이력 종류:

```text
입고
출고
입고 취소
출고 취소
검수
검수 정정
재검수
상태 변경
```

기준:

- 필터명을 추상적으로 쓰지 않는다.
- `현재 상태가 만들어진 과정 추적` 같은 모호한 필터명은 피한다.
- 필터는 실제 조회 조건으로 동작할 수 있어야 한다.

## 타임라인 항목

타임라인 항목에 포함할 정보:

- 이력 종류
- 관리번호
- 품목명
- 변경 전 상태
- 변경 후 상태
- 처리자
- 처리 시각
- 사유 또는 메모
- 연결 전표/검수 ID

기준:

- 상태 변화는 `before -> after` 형태로 보여준다.
- 중요한 상태는 badge로 표시한다.
- 처리 시각은 눈에 띄되 제목보다 강하면 안 된다.
- 긴 메모는 접거나 상세로 보낸다.
- 다음 처리 단계가 명확한 개별 타임라인은 해당 단계 안에 짧은 액션 버튼이나 차단 상태 안내를 둘 수 있다. 같은 행동을 하단 버튼으로 중복하지 않는다.

## 이력 타입 시각화

기준:

- 입고: blue
- 출고: navy 또는 gray
- 검수: green 또는 orange
- 불량/차단: red
- 취소/정정: orange
- 일반 상태 변경: gray

색상은 공통 토큰 안에서만 사용한다.

## 전체 이력 목록과 개별 타임라인

전체 이력 목록:

- 여러 부품의 이력을 검색/필터링한다.
- `data-table.md`의 테이블 기준을 함께 사용할 수 있다.

개별 타임라인:

- 한 개 관리번호의 흐름만 보여준다.
- 입고부터 현재 상태까지 순서가 명확해야 한다.
- 현재 상태 요약을 타임라인 상단에 둔다.

## 현재 상태 설명

개별 부품 상세 또는 이력 상세에서는 현재 상태를 문장으로 설명할 수 있다.

예:

```text
이 부품은 2026-05-18 입고 후 검수에서 B 등급을 받았고, 현재 판매 보류 상태입니다.
```

기준:

- 설명은 추론 가능한 사실만 사용한다.
- 추론과 저장된 이력을 혼동하지 않는다.
- 사유가 없는 상태 변경은 사유 없음으로 표시한다.

## 반응형 기준

- 모바일에서는 타임라인을 1열 카드로 표시한다.
- before/after 상태는 줄바꿈되어도 읽혀야 한다.
- 관리번호는 상단에 고정적으로 보이게 한다.
- 필터는 1열로 쌓는다.
