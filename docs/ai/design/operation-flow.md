# Operation Flow Design

전표 등록이나 검수처럼 여러 단계를 입력하고 최종 저장하는 업무 화면 기준이다.

목록은 `data-table.md`, 단순 드로어 폼은 `form-panel.md`, 오른쪽 안내는 `workflow-panel.md`를 따른다.

## 구조

```text
workspace-header
content-grid
- content-main
  - form.operation-flow-form
    - operation-step-card 반복
    - submit-panel
- side-panel
  - process-panel
```

- 본문 단계를 순서대로 세로 배치한다.
- 각 단계는 번호, 제목, 한 문장 설명, 입력 또는 검토 내용을 가진다.
- 오른쪽 패널은 입력이 아니라 현재 위치와 저장 후 흐름을 설명한다.
- 저장 전 확인이 필요하면 `modal-dialog.md`를 사용한다.

## 헤더

- 사용자가 새 업무를 작성 중임을 제목으로 명확히 한다.
- 보조 행동은 목록 복귀·취소, primary는 최종 저장이다.
- 헤더 저장과 마지막 저장 버튼은 같은 폼 동작을 호출한다.

## 단계 카드

```text
operation-step-card
- section-step
- title / description / status 선택
- fields, search, selection 또는 review
- field-message
```

- 단계 설명은 한 문장으로 제한한다.
- 필수 입력은 앞 단계에 모은다.
- 서버 자동 발급 값은 입력받지 않는다.
- 비어 있는 단계는 짧은 안내와 공통 empty state를 사용한다.

## 기준 데이터 검색과 선택

업무 중 거래처·품목·관리번호를 찾아야 하면 검색과 선택을 현재 흐름 안에서 처리한다.

- 검색 조건은 상단에 두고 좁은 폭에서 1열로 전환한다.
- 긴 검색 결과는 내부 스크롤을 허용한다.
- 선택 행은 공통 selected 상태를 사용한다.
- 선택한 대상과 다음 입력을 별도 요약 영역에서 연결한다.
- 기준 데이터가 없을 때만 최소 입력의 빠른 등록 모달을 제공한다.

## 라인 검토

- 전표에 추가한 품목·관리번호는 저장 전 별도 목록에서 검토한다.
- 같은 대상 합산, 수량·사유 수정, 제거 가능 여부는 feature 계약을 따른다.
- 빈 목록, 라인 수, 총수량을 명확히 표시한다.
- 카드 중첩보다 단순한 반복 행을 우선한다.

## 저장

```text
submit-panel
- 저장 후 생성·변경되는 핵심 상태
- submit-message
- form-actions
```

- 저장 후 전표, 재고, 관리번호, 이력 중 실제로 바뀌는 항목만 짧게 설명한다.
- 저장 중 중복 제출을 막는다.
- 성공 후 서버 응답 기준 화면으로 이동하거나 현재 화면을 갱신한다.
- 오류는 alert 대신 공통 피드백을 사용한다.

## 빠른 등록 모달

- 현재 작성 중인 내용을 잃지 않기 위한 보조 기능이다.
- 업무를 계속하는 데 필요한 최소 필드만 받는다.
- 전체 상세 입력은 해당 관리 화면으로 보낸다.
- 구조와 피드백은 `modal-dialog.md`를 따른다.

## 반응형

- 1180px 이하에서 workflow 패널을 본문 아래로 이동한다.
- 필드·검색·선택 영역은 좁은 폭에서 1열로 전환한다.
- 저장 버튼 텍스트가 줄바꿈되어도 높이를 유지한다.

## 금지

- 단계형 업무를 단순 목록이나 드로어에 수용
- workflow 패널에 실제 입력 추가
- 저장 후 상태 변화를 숨김
- 빠른 등록 모달에 전체 관리 기능 포함
- API 없이 화면 데이터만 성공 처리
