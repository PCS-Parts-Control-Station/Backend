# Public Pages Design

PCS 업무 공간 밖의 메인, 회사 등록, 로그인, 접근 오류 화면 기준이다. 토큰과 공통 요소는 `pcs-design-system.md`를 따른다.

## 대상

```text
/
/company/register
/w
/w/{companyCode}
/workspace-not-found
/page-not-found
/access-denied
```

업무 화면 레이아웃은 `workspace-layout.md`를 사용한다.

## 메인

목적은 과장된 마케팅보다 관리번호 기반 업무 흐름을 실제 운영 데이터로 이해시키는 것이다.

```text
Header
Hero
업무 화면 미리보기
현장 사용 사례
현장 문제와 핵심 기능
업무 공간 만들기
Footer
```

- Hero는 강제 100vh를 사용하지 않는다.
- 제목은 2줄까지, 설명은 운영 상황 중심으로 작성한다.
- CTA는 `회사 등록 시작`, `업체 로그인`을 기본으로 한다.
- 업체 코드 바로 접속은 보조 입력으로 제공할 수 있다.

업무 화면 미리보기에는 관리번호, 품목, 검수·판매 상태, 최근 이력처럼 실제 PCS 개념을 소량만 보여준다. 기능처럼 동작하는 가짜 입력은 만들지 않는다.

## 현장 사용 사례

- 구체적인 입고·검수·재고 상황을 짧은 카드로 보여준다.
- 자동 흐름을 사용하면 CSS 기반, 동일 묶음 1회 복제, hover/focus 정지, reduced motion 일반 grid를 적용한다.
- 읽기 어려운 속도, 갑작스러운 전환, 불필요한 JS를 사용하지 않는다.

## Full bleed

- 배경은 viewport 전체 폭, 콘텐츠는 공통 content width 안에 둔다.
- 한 페이지 1~3개로 제한한다.
- 소개 사례나 최종 CTA처럼 의미 있는 구분에만 사용한다.

## 회사 등록

소개보다 입력 작업을 우선한다.

```text
Header
짧은 page hero
register layout
- 회사·대표 연락처·OWNER 입력
- 등록 후 접속 정보 안내
completion state
```

- 입력 카드와 보조 패널의 상단 기준선을 맞춘다.
- 선택 연락처 때문에 필수 입력 흐름이 길어지지 않게 한다.
- 완료 후 자동 이동보다 회사명, 업체 코드, 접속 주소, OWNER loginId를 먼저 확인한다.
- 완료 상태는 주소 복사와 업체 로그인 이동을 제공한다.
- 진행 순서 카드와 설명을 과하게 추가하지 않는다.

## 로그인

`/w`:

- 업체 코드, 아이디, 비밀번호 입력

`/w/{companyCode}`:

- 업체 코드는 URL로 고정하고 아이디, 비밀번호만 입력
- 회사명과 업체 코드를 상단에 표시

공통:

- Header는 로고, 메인, 회사 등록만 둔다.
- Hero는 짧게 유지해 로그인 폼이 첫 화면에 보이게 한다.
- 데스크톱은 폼과 짧은 안내의 2열, 좁은 폭은 1열이다.
- 폼 안에 단계 번호와 중복 설명 제목을 넣지 않는다.
- 안내 패널은 사용자가 확인할 정보만 2~3개 둔다.
- API path, redirect URL, role code 같은 개발자 설명을 화면에 넣지 않는다.

## 접근 오류

오류 화면은 로그인과 같은 공개 페이지 톤을 사용한다.

- HTTP 상태, stack trace, 내부 예외명을 보여주지 않는다.
- 문제를 한 문장으로 설명하고 `업체 로그인`, `메인으로 이동`을 제공한다.
- 업체 코드가 있으면 작은 code chip으로 표시한다.
- 확인 항목은 3~4개 이하이며 첫 화면에 핵심 문구와 버튼이 보여야 한다.
- 보조 패널의 위아래 여백과 카드 높이를 일정하게 유지한다.

## Motion

- 소개 화면은 500ms 안팎의 작은 reveal을 사용할 수 있다.
- reduced motion에서는 제거한다.
- 큰 scale, 회전, 과한 parallax를 사용하지 않는다.

## 금지

- 로그인·등록 화면에 긴 마케팅 섹션 추가
- 사용자 화면에 API·권한 구현 설명 노출
- 읽기 어려운 자동 슬라이드
- 고정 높이로 작은 화면 콘텐츠 잘림
