# Public Pages Design

PCS 업무공간 밖의 공개/진입 화면 기준이다.

## 적용 대상

- `/`
- `/company/register`
- 회사 등록 완료 상태
- `/w`
- `/w/{companyCode}`
- `/workspace-not-found`
- `/page-not-found`
- `/access-denied`

## 적용 제외

- 로그인 후 업무 관리 화면
- 데이터 목록/테이블 화면
- 상세/이력 화면

## 메인 페이지 방향

메인 화면은 단순한 관리자 페이지보다 한 단계 더 소개성이 있어야 한다. 하지만 과장된 마케팅 페이지처럼 보이면 안 된다.

핵심은 아래 흐름이 한눈에 이해되는 B2B SaaS 스타일이다.

```text
관리번호 기반 개별 부품 추적
입고
검수
재고
출고
이력
```

메인 화면에서는 `관리번호`, `검수등급`, `검수상태`, `판매상태`, `보관위치`, `최근이력` 같은 실제 운영 데이터를 보여준다.

## 메인 페이지 구조

현재 `/` 메인 페이지의 섹션 흐름:

```text
Header
Hero
관리자 화면 미리보기
현장 사용 사례
현장에서 자주 생기는 문제
PCS 핵심 기능
업무공간 만들기
Footer
```

역할:

- `Hero`: 서비스 정체성과 핵심 메시지 전달
- `관리자 화면 미리보기`: PCS가 실제 업무 시스템이라는 인상 제공
- `현장 사용 사례`: PCS가 쓰이는 구체적인 현장 설명
- `현장에서 자주 생기는 문제`: 도입 필요성 제기
- `PCS 핵심 기능`: 문제에 대한 해결 방식 설명
- `업무공간 만들기`: 최종 행동 유도

## 메인 Hero

기준:

- Hero는 이미 충분히 크므로 강제로 `100vh`를 만들지 않는다.
- 제목은 중앙 정렬하고 2줄 구성이 가능하다.
- 설명문은 기능 나열보다 운영 상황을 설명한다.
- CTA는 `회사 등록 시작`, `업체 로그인`을 기본 쌍으로 둔다.
- 업체 코드 바로 접속 폼은 Hero 하단에 둘 수 있다.

메인 Hero 제목 기준:

```css
h1 {
    max-width: 900px;
    margin: 0 auto;
    font-size: clamp(42px, 5.7vw, 68px);
    font-weight: 800;
    line-height: 1.1;
}
```

## 관리자 화면 미리보기

메인 화면의 미리보기는 실제 PCS 관리자 화면처럼 보여야 한다.

포함 정보:

- 관리번호
- 품목명
- 검수등급
- 검수상태
- 판매상태
- 보관위치
- 최근이력

미리보기 구조:

```text
상단 툴바
사이드 메뉴
요약 숫자
관리번호 박스
상태 테이블
```

사이드 메뉴 라벨은 한국어 업무 용어를 사용한다.

```text
운영 현황
품목 관리
입고 등록
검수 관리
출고 등록
전표 통합 조회
```

미리보기는 실제 기능이 아니라 메인 화면의 제품 이해를 돕는 시각 자료다. 너무 많은 데이터를 넣지 말고 PCS의 핵심인 `개별 부품 상태 추적`만 드러낸다.

## 풀블리드 섹션

메인 화면에는 중간중간 화면 양옆을 꽉 채우는 풀블리드 밴드를 사용한다.

현재 풀블리드 대상:

- `현장 사용 사례`
- `업무공간 만들기`

기본 구조:

```html
<section class="full-bleed-section">
    <div class="section-inner">
        ...
    </div>
</section>
```

기준:

- 배경은 화면 전체 폭을 채운다.
- 텍스트와 버튼은 중앙 콘텐츠 폭 안에 둔다.
- 너무 많은 섹션을 풀블리드로 만들지 않는다.
- 한 페이지에서 1~3개 정도만 사용한다.

## 현장 사용 사례 카드

현장 사용 사례는 자동으로 흐르는 카드 트랙을 사용할 수 있다.

기준:

- CSS 기반 marquee 애니메이션
- 같은 카드 묶음을 한 번 복제해 끊김 없이 반복
- hover 또는 focus 시 애니메이션 일시 정지
- `prefers-reduced-motion: reduce`에서는 애니메이션 제거 후 일반 그리드로 표시

카드 예시:

```text
PC방 일괄 매입
RAM, SSD, GPU가 한 번에 들어오는 날
거래처 전표로 묶고, 부품마다 관리번호를 발급해 검수 대기와 판매 가능 수량을 나눠 봅니다.
```

금지:

- 빠르게 튀는 슬라이드
- 카드가 갑자기 사라지는 전환
- 사용자가 읽기 어려울 정도로 빠른 속도
- JS가 꼭 필요하지 않은 단순 애니메이션에 JS 추가

## 회사 등록 페이지

`/company/register`는 사용자가 정보를 입력하는 화면이므로 메인 페이지보다 소개성을 낮추고 입력 작업이 중심이 되게 만든다.

기본 흐름:

```text
Header
Page Hero
Register Layout
Form Sections
Completion State
```

회사 등록 페이지 기준:

```text
Header
- 로고
- 메인
- 업체 로그인

Page Hero
- 짧은 eyebrow
- 짧은 제목
- 한 줄 수준의 설명

Register Layout
- 왼쪽: 입력 폼 카드
- 오른쪽: 등록 후 접속 정보 패널

Form Sections
- 회사 정보
- 대표 연락처
- 최고 관리자 계정

Completion State
- 회사명
- 업체 코드
- 업체 접속 주소
- Owner 로그인 ID
- 주소 복사
- 업체 로그인 화면 이동
```

기준:

- 상단 제목과 폼 카드는 같은 기준선에 맞춘다.
- 폼 카드 폭은 입력 구조에 맞게 잡고 불필요한 빈 공간이 크게 남지 않게 한다.
- 오른쪽 보조 패널은 `접속 주소`, `회사명`, `Owner 로그인 ID`, 짧은 안내문 정도만 둔다.
- 등록 완료 후에는 자동 이동보다 완료 화면에서 업체 접속 주소를 먼저 확인하게 한다.
- 진행 순서 박스나 설명 카드를 과하게 넣어 폼 입력을 방해하지 않는다.

## 로그인 페이지

`/w`:

- 업체 코드 + 아이디 + 비밀번호를 입력한다.
- 업체 코드를 모르는 사용자를 위한 기본 진입점이다.

`/w/{companyCode}`:

- URL로 업체 코드가 정해졌으므로 아이디 + 비밀번호만 입력한다.
- 업체 코드와 접속 중인 회사명을 상단에 명확히 보여준다.

로그인 화면은 마케팅성 섹션을 넣지 않고, 빠르게 업무공간에 들어가는 것이 목적이다.

현재 로그인 페이지 기준:

```text
Header
Page Hero
Login Layout
- 왼쪽: 로그인 입력 폼
- 오른쪽: 짧은 로그인 안내 패널
```

Header:

- 로고, 메인, 회사 등록 링크만 둔다.
- 업무 화면 메뉴나 개발용 경로 설명은 넣지 않는다.

Page Hero:

- 짧은 eyebrow, 큰 제목, 한 줄 수준의 설명으로 구성한다.
- 메인 페이지처럼 소개성 섹션을 길게 만들지 않는다.
- 로그인 카드가 첫 화면에서 바로 보이도록 Hero 설명은 짧게 유지한다.

Login Layout:

- 데스크톱에서는 왼쪽 폼과 오른쪽 안내 패널을 2열로 둔다.
- 왼쪽 폼은 오른쪽 안내 패널보다 주요 영역이므로 폭을 더 크게 둔다.
- 오른쪽 안내 패널은 보조 정보만 담고, 시선이 입력 폼보다 강하게 튀면 안 된다.
- 로그인 카드 기본 스케일은 `width: min(960px, 100%)`, 오른쪽 패널 `420px`, 입력 컨트롤 높이 `48px`를 기준으로 한다.

로그인 입력 폼:

- 폼 내부에 `로그인 정보`, 단계 번호, 별도 설명 제목을 넣지 않는다.
- 첫 입력 필드는 카드 상단에 너무 붙이지 않고 충분한 위 여백을 둔다.
- 버튼 아래쪽 빈 공간은 과하게 남기지 않는다.
- `/w`에서는 `업체 코드`, `아이디`, `비밀번호`를 입력한다.
- `/w/{companyCode}`에서는 업체 코드 입력칸을 숨기고, 접속 중인 업체 코드를 chip으로 보여준다.

오른쪽 로그인 안내 패널:

- `로그인 안내`, `업체 선택 필요`, `현재 접속 방식`, `로그인 후 이동` 같은 보조 라벨을 남발하지 않는다.
- 안내 패널은 제목과 짧은 카드 2개, 하단 확인 문구 정도로만 구성한다.
- 카드 안에는 사용자가 바로 이해할 업무 문구만 넣는다.
- `POST /api/...`, 성공 후 이동 URL, 내부 API 이름처럼 개발자에게 필요한 정보는 사용자 화면에 넣지 않는다.
- 하단 확인 문구 아래쪽 여백은 상단 여백과 균형을 맞춘다.

현재 권장 문구:

```text
필요한 정보만 입력하면 됩니다.
업체 코드를 입력해야 합니다.
회사 관리자에게 받은 업체 코드, 아이디, 비밀번호를 입력하세요.
업무 대시보드
부품 재고, 입고, 검수, 출고, 이력 관리 메뉴를 이어서 사용할 수 있습니다.
로그인이 안 되나요?
업체 코드, 아이디, 임시 비밀번호는 회사 관리자에게 확인하세요.
```

## 스크롤 동작

메인 페이지와 소개성 화면은 과하지 않은 reveal 동작을 사용할 수 있다.

기준:

- 스크롤 시 섹션 제목과 카드가 살짝 아래에서 올라오며 나타난다.
- 움직임은 500ms 안팎으로 짧게 유지한다.
- `prefers-reduced-motion: reduce` 환경에서는 애니메이션을 끈다.

금지:

- 큰 scale 애니메이션
- 회전, 과한 parallax
- 카드가 튀는 듯한 움직임

## 잘못된 접근 안내 페이지

`/workspace-not-found`, `/page-not-found`, `/access-denied`는 같은 정적 HTML을 사용해도 된다.

기준:

- 로그인/회사 등록 화면과 같은 공개 페이지 톤을 사용한다.
- 카드 폭, 오른쪽 패널 폭, 제목 크기, 버튼 높이는 로그인 페이지 스케일을 따른다.
- 데스크톱 기준 카드는 `width: min(960px, 100%)`, 오른쪽 패널은 `420px`, 제목은 `28~36px` 범위로 둔다.
- 세로 높이는 고정하지 않고 내용 높이에 맞춘다.
- 개발자용 HTTP 상태, API 경로, stack trace, 내부 예외명은 보여주지 않는다.
- 사용자가 다음 행동을 바로 고를 수 있게 `업체 로그인`, `메인으로 이동` 버튼을 제공한다.
- 업체 코드가 있으면 작은 코드 박스로 보여준다.
- 오른쪽 보조 패널에는 확인할 내용 3~4개만 둔다.
- 오른쪽 보조 패널의 콘텐츠는 위아래 공백이 같아 보이도록 세로 중앙 정렬한다.
- 제목에 임의의 큰 `margin-top`을 주어 위쪽 또는 아래쪽 공백이 한쪽으로 쏠리게 만들지 않는다.
- 확인 항목 카드는 같은 높이와 같은 간격을 유지해 패널 안에서 자연스럽게 묶여 보이게 한다.
- 화면은 첫 진입에서 핵심 문구와 버튼이 보이도록 과하게 길게 만들지 않는다.

권장 문구:

```text
업체 주소를 확인할 수 없습니다.
입력한 업체 코드가 잘못되었거나 더 이상 사용할 수 없는 주소입니다.
회사 관리자에게 받은 업체 코드가 맞는지 확인하세요.
로그인 후에도 같은 화면이 나오면 계정 권한을 확인하세요.
```
