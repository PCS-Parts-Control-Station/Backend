# PCS Permission Rules

이 문서는 PCS 권한 기준의 원본 문서다.
기능 문서에서는 권한을 다시 길게 정의하지 않고 이 문서를 참조한다.

## 기본 관점

PCS의 권한은 업무를 막기 위한 등급표가 아니다.

이 프로젝트에서 실제 입고, 검수, 출고, 품목, 품목 분류, 사양 항목, 거래처 관리는 현장 직원이 처리하는 일상 업무다.
따라서 `STAFF`도 업무 관련 데이터 생성, 수정, 상태 변경을 기본적으로 수행할 수 있어야 한다.

권한의 핵심 차이는 아래처럼 본다.

```text
OWNER = 회사 소유/복구 계정
ADMIN = STAFF 업무 가능 + 사용자 관리
STAFF = 실무 대부분 처리
```

## 역할

```text
OWNER
ADMIN
STAFF
```

## OWNER

업체 작업공간의 최상위 소유자 계정이다.

주요 성격:

- 회사 생성 후 만들어지는 최고 권한 계정
- 회사 정보와 업체 작업공간의 최종 소유자
- ADMIN 계정 생성과 복구를 담당할 수 있는 계정
- 일상 업무 계정보다는 복구/최종 관리 계정에 가깝다

가능한 일:

- 회사 정보 조회/수정/활성 여부 변경
- 사용자 관리
- ADMIN/STAFF 계정 생성, 수정, 권한 변경, 임시 비밀번호 발급
- 업무 관련 모든 기능 사용
- 내 정보 관리

## ADMIN

업체 내부의 계정 관리자다.

주요 성격:

- STAFF와 동일하게 실무 기능을 사용할 수 있다.
- 추가로 사용자 계정을 관리한다.
- PCS에서 ADMIN은 품목/거래처/입출고 업무를 승인하는 별도 결재자가 아니다.

가능한 일:

- 사용자 목록 조회
- STAFF 계정 생성/수정
- STAFF 임시 비밀번호 발급
- STAFF 계정 사용 여부 변경
- 업무 관련 모든 기능 사용
- 내 정보 관리

제한:

- OWNER 계정의 권한을 낮추거나 비활성화하지 않는다.
- 회사 소유권 성격의 설정은 OWNER 기준을 따른다.

## STAFF

실제 업무를 처리하는 실무 계정이다.

주요 성격:

- PCS에서 가장 자주 사용하는 계정이다.
- 입고 중 새 거래처나 새 품목이 나오면 바로 등록할 수 있어야 한다.
- 품목 분류와 사양 항목도 업무 기준이므로 필요할 때 직접 관리할 수 있어야 한다.

가능한 일:

- 대시보드 조회
- 품목 관리
- 품목 분류
- 사양 항목 관리
- 거래처 관리
- 입고 등록/취소
- 검수 등록/정정/재검수
- 출고 등록/취소
- 재고 조회
- 이력 조회
- 내 정보 관리

제한:

- 사용자 생성/수정/권한 변경/임시 비밀번호 발급은 할 수 없다.
- 회사 정보 변경은 할 수 없다.
- OWNER/ADMIN 계정 관리에는 접근하지 않는다.

## 기능별 기준

| 기능 영역 | OWNER | ADMIN | STAFF |
|---|---:|---:|---:|
| 회사 생성 | 가능 | 불가 | 불가 |
| 회사 정보 조회/수정 | 가능 | 제한 | 불가 |
| 사용자 목록/생성/수정 | 가능 | 가능 | 불가 |
| 사용자 권한 변경 | 가능 | 가능 | 불가 |
| 임시 비밀번호 발급 | 가능 | 가능 | 불가 |
| 내 정보 조회/수정 | 가능 | 가능 | 가능 |
| 대시보드 | 가능 | 가능 | 가능 |
| 품목 관리 | 가능 | 가능 | 가능 |
| 품목 분류 | 가능 | 가능 | 가능 |
| 사양 항목 관리 | 가능 | 가능 | 가능 |
| 거래처 관리 | 가능 | 가능 | 가능 |
| 입고 | 가능 | 가능 | 가능 |
| 검수 | 가능 | 가능 | 가능 |
| 출고 | 가능 | 가능 | 가능 |
| 재고 조회 | 가능 | 가능 | 가능 |
| 이력 조회 | 가능 | 가능 | 가능 |
| 이력 수정/삭제 | 불가 | 불가 | 불가 |

`제한`은 기능을 바로 열지 않고 별도 정책이 있을 때만 허용한다는 뜻이다.
초기 기준에서 회사 정보 변경은 OWNER 중심으로 본다.

## STAFF 업무 권한 설정

STAFF 업무 권한은 개인별로 세밀하게 나누기보다 업체 단위 정책으로 먼저 다룬다.

기준:

- 기본값은 STAFF 업무 권한 전체 허용이다.
- OWNER와 ADMIN은 사용자 관리 화면에서 STAFF 공통 업무 권한을 조정할 수 있다.
- 이 설정은 STAFF에게만 적용하며 OWNER/ADMIN의 업무 권한에는 영향을 주지 않는다.
- 권한을 끄면 해당 업체의 모든 STAFF가 그 업무 메뉴와 API를 사용할 수 없다.
- 권한 설정은 꺼진 권한만 `tb_company_staff_permission_disabled`에 저장한다.
- 개인별 예외 권한은 초기 범위가 아니며, 필요해지면 별도 정책과 테이블로 분리한다.

초기 STAFF 업무 권한:

```text
STAFF_PARTNER_MANAGE
STAFF_PART_CREATE
STAFF_CATEGORY_MANAGE
STAFF_INBOUND
STAFF_INSPECTION
STAFF_OUTBOUND
```

품목 관련 메뉴 표시 규칙:

- 사이드바에는 `부품 관리`를 표시하고 `품목 관리`, `품목 분류`는 별도 메뉴로 두지 않는다.
- 품목 관리는 부품 관리 화면 또는 관련 업무 화면의 `품목 관리` 버튼으로 이동한다.
- 품목 분류는 품목 관리 화면의 `품목 분류` 버튼으로 이동한다.
- 품목 관리 화면의 `품목 분류` 버튼은 `STAFF_CATEGORY_MANAGE`가 있을 때만 표시한다.

## 삭제 / 사용 중지 / 취소 기준

권한만으로 삭제 가능 여부를 판단하지 않는다.

기준:

- 하드 삭제는 초기 업무 기능의 기본 흐름으로 보지 않는다.
- 삭제 API가 있는 기능은 연결 데이터가 없을 때만 허용한다.
- 연결 데이터가 있으면 이름/설명/상태 정리로 처리한다.
- 이력 데이터는 수정/삭제하지 않고 새 이력으로 남긴다.
- 입고/출고 취소처럼 업무상 되돌리기가 필요한 기능은 취소 이력을 남기는 방식으로 처리한다.

예:

- 품목 분류 삭제는 연결된 품목이 없을 때만 가능하다.
- 거래처는 삭제보다 거래 가능 여부 변경을 우선한다.
- 입출고 전표는 삭제가 아니라 취소 전표 또는 취소 이력으로 남긴다.

## 기본 인가 규칙

업무 API의 기본값:

```text
/api/workspaces/{companyCode}/** 업무 데이터 API
-> 인증된 OWNER / ADMIN / STAFF 모두 접근 가능
```

별도 제한이 필요한 API만 예외를 둔다.

예외:

```text
회사 소유권/회사 정보 변경
-> OWNER 중심

사용자 관리
-> OWNER / ADMIN

OWNER 계정 관리
-> OWNER

이력 수정/삭제
-> 허용하지 않음
```

## 구현 기준

- Controller에서 문자열로 권한을 직접 판단하지 않는다.
- 인증 사용자는 `PcsPrincipal`에서 받고, role 값은 프로젝트 Enum 기준으로 검증한다.
- role 비교는 `MemberRole` enum 기준으로 처리한다.
- Spring Security URL 접근 제어는 `SecurityConfig`에서 처리하고, role 그룹은 `PcsRoleGroups`를 사용한다.
- 회사 범위 검증은 `docs/features/auth.md`의 `companyCode`, `companyId` 기준을 따른다.
- `/api/workspaces/{companyCode}/**` API는 권한보다 먼저 회사 범위와 계정 활성 여부를 검증한다.
- 기능 문서가 권한 기준을 적어야 하면 이 문서를 참조하고, 해당 기능에서 특별한 제한만 추가로 적는다.

## 문서 작성 기준

새 기능 문서에서 권한을 적을 때는 아래처럼 쓴다.

```text
- 권한 기준은 docs/ai/pcs-permission-rules.md를 따른다.
- 이 기능은 별도 권한 제한이 없으므로 인증된 OWNER/ADMIN/STAFF 모두 사용할 수 있다.
```

특별 제한이 있으면 예외만 적는다.

```text
- 사용자 임시 비밀번호 발급은 OWNER/ADMIN만 가능하다.
```

기능 문서마다 OWNER/ADMIN/STAFF 전체 표를 다시 만들지 않는다.
